# DR-002 - Drop "Require a pull request before merging" branch protection

DR-ID: DR-002-20260428-195700
Date: 2026-04-28 19:57:00
Asked by: stevegt+ppx@t7a.org (stevegt-via-perplexity)
State: decided
Question: Should the `main` branch ruleset on github.com/promisegrid/wire-lab keep the GitHub-specific "Require a pull request before merging" rule, or drop it in favor of a Git-native "only Steve can push to main" model?
Why this blocks progress: PromiseGrid is the long-term replacement for GitHub. Any workflow that depends on GitHub-specific PR semantics ties the repo's coordination model to GitHub and creates migration debt. Deciding this up front lets the bot's task workflow (DR-003) be designed against the correct merge mechanism.
Affects: github.com/promisegrid/wire-lab branch ruleset on `main`; how Steve performs merges; whether the bot ever opens GitHub PRs; the migration story to PromiseGrid; what other contributors (if any) will have to do.
Unblocks: TODO/001-perplexity-computer-onboarding.md (subtask 001.2 and 001.4); DR-003 review style; all future bot branches.
Waiting on: stevegt@t7a.org (Steve Traugott)

## Candidate alternatives considered

- (a) Drop "Require a pull request before merging". Keep "Restrict who can push to matching branches → stevegt only", "Block force pushes", "Restrict deletions". Steve merges by direct push to `main`; no GitHub PRs needed. Selected.
- (b) Keep "Require a pull request before merging" but treat the PR as a thin merge-button transport whose body just points at the DR. Adds GitHub-specific friction.
- (c) Drop "Require PR" for now, revisit if more contributors join. Same effect as (a) but with explicit revisit trigger; not chosen because (a) doesn't preclude revisiting either.

## Decision

(a). Drop the "Require a pull request before merging" rule. Keep "Restrict who can push to matching branches → stevegt only", plus "Block force pushes" and "Restrict deletions". Steve merges by direct push to `main`; the bot pushes only to `ppx/{twig}` branches and never to `main`.

The "only Steve can push to main" rule maps cleanly to PromiseGrid's eventual "the canonical pointer follows Steve's signing key" semantics in `specs/harness-spec-draft.md` §10a.8 — the same constraint expressed in a forge-specific rule today, and a Git-native (then PromiseGrid-native) constraint tomorrow.
