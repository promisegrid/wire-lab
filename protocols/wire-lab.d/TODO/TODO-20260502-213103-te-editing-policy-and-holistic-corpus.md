### DI-020-20260502-232651 (DF-35.1 -> Alt-1.C, supersedes DI-020-20260502-213103's Cat-1 clause)

I promise that Cat-1 path renames in the TE editing policy are split into two subcategories, Cat-1a and Cat-1b, classified per match during a sweep.

**Cat-1a (current-pointer paths)** are references whose meaning is "the file currently lives at X." These are mechanically swept in place when a path migrates: the old path string is rewritten to the new path string, no top-of-file note is required because the rewrite is self-explanatory in `git log -p`. The vast majority of path references in the corpus are Cat-1a.

**Cat-1b (historical-quotation paths)** are references whose meaning is "TE-M, written on date D, said the path was X" or "the corpus at time T showed path X." These are left untouched by sweeps. If a path migrates, Cat-1b references continue to show the old string because that is what the historical record said at the time the citing TE was written. Rewriting a Cat-1b reference would silently falsify the quotation and degrade the corpus's self-evidence — the property that DI-020-20260502-213105 (holistic reading for substantive questions) depends on, because holistic reading depends on cross-TE quotations being trustworthy.

Classification heuristics for sweeps (apply per match):

1. If the path appears inside a quotation context — a markdown blockquote (`> ...`), a fenced code block presented as a citation, or text introduced by phrasing like "TE-N says," "the corpus shows," "as of timestamp T," "originally read," "at the time of writing" — it is Cat-1b. Leave it.
2. If the path appears inside a `## Refinements` section, a supersedence note, a `Decision status` line, or any other historical-record framing, it is Cat-1b. Leave it.
3. If the path appears in a sentence whose tense is past or perfect with respect to the corpus state ("TE-30 referenced X"), it is Cat-1b. Leave it.
4. Otherwise it is Cat-1a. Sweep it.
5. When in doubt, treat as Cat-1b and leave it. False negatives (a Cat-1a left unswept) produce a single stale string that is easy to find and fix later; false positives (a Cat-1b swept) silently falsify the historical record and are not recoverable without git archaeology.

A Cat-1 sweep is therefore no longer a one-line `sed -i`; it is a careful read-through of every match with per-match classification. The cost is real (sweeps slow down) but bounded, and the alternative is accepting silent corpus degradation. Tooling may assist (a script that emits each match with surrounding context for human review is acceptable; a script that auto-rewrites without review is not).

This DI was reached via TE-35 Scenario S3 (Dave's mid-twig path migration) and reinforced by S5 / S6 (Mallory bad-faith Cat-2 attack and verbatim cross-TE quotation). TE-30, TE-32, and TE-33 in the existing corpus contain Cat-1b references; subtask 020.6 (TE path-reference sweep) must execute under this refined rule, not the original DI-020-20260502-213103 Cat-1 clause.

DI-020-20260502-213103 otherwise stands. The Cat-2 / Cat-3 / Cat-4 / Cat-5 / Cat-6 / Cat-7 clauses are unchanged. Only the Cat-1 clause is superseded by this DI.

### DI-020-20260502-213103 (DF-34.1 -> Alt-1.C)

I promise that TE editing in the wire-lab repo follows a categorized policy after first commit. Mechanical edits (Cat-1 path renames, Cat-2 vocabulary updates) are made in place. Cat-2 vocabulary updates carry a top-of-file note pointing at the TE or TODO that drove the rewrite, following the existing TE-24 / TE-26 precedent; Cat-1 path renames need no such note because the diff is self-explanatory in `git log -p`. Navigational edits (Cat-3 forward-pointers to later TEs / DRs / DIs that resolved a future-work bullet, and Cat-4 back-pointers added after the fact) go in an append-only `## Refinements` section at the bottom of the TE; the original body is not modified. Substantive edits (Cat-5 recasting alternative analysis, Cat-6 reversing a locked decision, Cat-7 invalidating an assumption) are not edits at all: a new TE that supersedes the old one is written, a new DI that supersedes the old DI is written, and the only change to the original TE is a one-line update to its `Decision status` line pointing at the supersedence. TE filenames remain immutable per TE-25's invariant.

This DI retires the previously-claimed "do not back-edit TEs" rule. That rule was never actually written in any AGENTS file or repo doc; it was a bot overgeneralization from DI-003 (contest-artifact immutability for `proposals/pending/<branch>/contest-*.md` and `review-*.md`). DI-003 still stands for adversarial review records; it does not transfer to TEs, which are the author's own thinking and have always been editable in the TE-24 / TE-26 sense.

### DI-020-20260502-213104 (DF-34.2 -> Alt-2.A)

I promise that the TE editing policy locked in DI-020-20260502-213103 applies uniformly across all TE corpora in the wire-lab repository, including the harness corpus at `docs/thought-experiments/` and any future per-protocol corpora at `protocols/<slug>.d/docs/thought-experiments/`. There is one editing policy, not one per protocol. A future protocol that judges its threat model demands a stricter regime (for example, a security-sensitive protocol whose TEs may be cited as evidence in dispute resolution) may write its own superseding TE that locks a different policy for that corpus only; absent such a superseding TE, this policy is in force.

This is a deliberate departure from TE-34's recommended Alt-2.B (per-protocol). The reasoning for choosing Alt-2.A: the per-protocol simrepos are still hypothetical at this point in the repo's life, and predicting which of them will need a different editing policy is speculation. Locking one policy now, with the explicit affordance that any protocol may override by writing its own superseding TE, is operationally simpler and matches the user's preference. The "let each protocol name its own internals" principle (TE-23 DI-23.2) is preserved by the override path; it is just not the default.

### DI-020-20260502-213105 (DF-34.3 -> Alt-3.C)

I promise that the bot reads the TE corpus with an explicit split: holistic-by-default for substantive questions, single-TE for obviously mechanical ones. When any TE is in scope and the question touches design, framing, decisions, alternatives, or tradeoffs, the bot's first move is to scan TE titles and "Decision under test" sections across the entire corpus, identify TEs that share assumptions, vocabulary, or decisions with the one in scope, and deep-read those neighbors selectively before answering. When the question is obviously mechanical (for example, "what is TE-N's filename?", "what timestamp does TE-N have?", "does TE-N reference file X?"), the bot reads only the referenced TE.

This is a deliberate departure from TE-34's recommended Alt-3.B (holistic-by-default with Alt-3.C as the practical implementation). The reasoning for choosing Alt-3.C explicitly: naming the narrowing rule upfront makes it auditable. A reader of AGENTS.md should be able to predict whether a question will trigger a holistic scan; under Alt-3.B that prediction depends on the bot's judgment about whether a question is "obviously mechanical," which is the same judgment Alt-3.C names explicitly. The classification task itself remains a small holistic-reading task (you have to know what a question touches to know whether it is mechanical), but having the rule say so out loud is better than implying it.

For the wire-lab corpus today (TE-1 through TE-34), this means: any question about decisions, framing, locked DIs, alternative analysis, future-work bullets, or design tradeoffs triggers a corpus scan; questions about file paths, filenames, timestamps, or single-line lookups inside one TE do not.
